# 🍦 Melt Risk Agent

An end-to-end **Data Engineering + GenAI + Agentic AI** project: an AI analyst that monitors
an ice cream cold-chain distribution network and answers questions like *"which shipments
are at risk of melting?"* or *"what's understocked?"* — grounded in real pipeline data, not
hallucinated numbers.

Built entirely with **free, open-source tools** — no paid APIs, no cloud trial that expires.

---

## Why this project (and why ice cream)

Cold-chain / perishable-goods distribution is a genuinely harder supply chain problem than
generic retail: a shipment isn't just "late," a delay past a product's **melt tolerance
window** turns it into a write-off. That constraint gives the GenAI agent something real to
reason about, instead of just reading numbers back to you.

## Architecture

```
Synthetic Data Generator (Python + Faker)
        |
        v
   Raw CSVs  --------------------->  DuckDB (raw schema)
        (ingestion / landing zone, same role as ADF / Fivetran)
        |
        v
   dbt (staging models: typed, cleaned views)
        |
        v
   dbt (mart models: business logic)
        - mart_melt_risk_shipments  (delay vs. melt tolerance -> risk level)
        - mart_inventory_health     (stock-out + warm-freezer risk)
        - mart_demand_trends        (daily demand, feeds forecasting)
        |
        v
   Agentic AI layer (Groq: Llama 3.3 70B, tool-calling loop)
        PLAN -> ACT (run SQL / call tools) -> OBSERVE -> RESPOND
        |
        v
   Streamlit chat UI  (deployed free on Streamlit Community Cloud)
```

**Tech stack:** Python, DuckDB, dbt-core + dbt-duckdb, Groq API (free tier), Streamlit.
Every piece maps to an enterprise equivalent: DuckDB ≈ Snowflake (same SQL-first warehouse
concept), the ingestion step ≈ Azure Data Factory / Fivetran, dbt is literally the same tool
used in production data teams, and the agent's tool-calling loop is the same pattern used in
production LLM agent systems (plan → act → observe → respond).

## What makes this "agentic" and not just "a chatbot with an LLM"

The agent doesn't just answer from its own knowledge. Each turn, the LLM decides **which
tool(s)** it needs (query the warehouse directly with SQL, or call a purpose-built tool like
`get_melt_risk_shipments` or `simple_demand_forecast`), the tool actually executes against
the real DuckDB warehouse, the result is fed back to the LLM, and only then does it respond.
This is a genuine plan → act → observe → respond loop, not a single completion call.

## Project structure

```

│   ├── generate_data.py      # synthetic cold-chain data generator
│   ├── load_to_duckdb.py     # ingestion: raw CSVs -> DuckDB

│   ├── dbt_project.yml

├── requirements.txt
└── .streamlit/secrets.toml.example
```

## Running it locally (5 minutes)

1. **Clone and install:**
   ```bash
   git clone <your-repo-url>
   cd melt-risk-agent
   pip install -r requirements.txt
   ```

2. **Get a free Groq API key:** https://console.groq.com/keys (no credit card required)

3. **Run the pipeline** (generates data, loads DuckDB, runs dbt):
   ```bash
   bash run_pipeline.sh
   ```

4. **Launch the app:**
   ```bash
   export GROQ_API_KEY="your-key-here"
   streamlit run app/streamlit_app.py
   ```
   Or skip the export and just paste the key into the app's UI on first load.

## Deploying for free (Streamlit Community Cloud)

1. Push this repo to GitHub (see checklist below).
2. Go to https://share.streamlit.io → "New app" → point it at your repo, branch `main`,
   main file path `app/streamlit_app.py`.
3. In the app's **Settings → Secrets**, paste:
   ```toml
   GROQ_API_KEY = "your-key-here"
   ```
4. Deploy. The app **self-bootstraps the entire pipeline** on first load (see
   `bootstrap_pipeline()` in `streamlit_app.py`) — no manual data setup needed on the
   deployed instance.

### Before pushing to GitHub
- Make sure `.streamlit/secrets.toml` (the real one, if you made one locally) is **not**
  committed — it's already in `.gitignore`.
- Never commit your actual Groq API key anywhere in code or commit history.

## Talking points for interviews

- **Data engineering fundamentals**: layered warehouse design (raw → staging → marts),
  source-to-model lineage via dbt, idempotent transformations, portable environment config.
- **GenAI**: natural-language-to-SQL-and-back, grounding LLM responses in real query results
  rather than letting it hallucinate.
- **Agentic AI**: a genuine multi-step tool-calling loop (not a single prompt), tool design
  (when to expose raw SQL access vs. a purpose-built function), and reasoning over multiple
  tool calls before responding.
- **Why DuckDB instead of Snowflake**: same SQL semantics and warehouse mental model, but
  free and local — the dbt models here would need almost no changes to run against Snowflake
  in production (swap the adapter and connection profile).

## Extending this further

- Add a `dbt test` suite (not-null, accepted values, relationships) to show data quality
  rigor.
- Swap the naive moving-average forecast for a proper time-series model (e.g. Prophet or a
  simple ARIMA) — same tool interface, smarter implementation.
- Add a second agent tool that calls a free weather API, since ice cream demand is genuinely
  weather-driven — a nice real "multi-tool" agent demo.
- Orchestrate `run_pipeline.sh` on a schedule with GitHub Actions (free) to simulate a daily
  batch job instead of running it manually.
